Capítulo 1
O que é uma API?

API significa Application Programming Interface — interface de programação de aplicações. É um contrato que define como dois sistemas de software se comunicam: o que pode ser pedido, como pedir, e o que se recebe de volta.

Pense em um restaurante: o cardápio é a "documentação da API", o garçom é a "API" que recebe seu pedido, leva para a cozinha e traz o resultado. Você não precisa saber como a comida é preparada — só precisa saber o que pedir e o que esperar receber.

Por que APIs existem?

  • Separação de responsabilidades — frontend e backend evoluem de forma independente
  • Reutilização — o mesmo backend atende app mobile, web e outros serviços
  • Integração — sistemas diferentes conversam por um protocolo comum
  • Segurança — você expõe só o que precisa, sem dar acesso direto ao banco de dados

Tipos de API

TipoDescriçãoUso comum
RESTUsa HTTP + recursos na URLWeb, apps móveis — o mais usado
GraphQLQuery flexível em único endpointApps com dados complexos e variáveis
gRPCBinário, alto desempenhoComunicação interna entre microsserviços
WebSocketConexão persistente bidirecionalChat, notificações em tempo real
SOAPXML sobre HTTP, formal/legadoSistemas bancários e governamentais antigos
ℹ️
Foco deste guia

Vamos trabalhar exclusivamente com REST APIs sobre HTTP — o formato mais utilizado no mercado e o adotado pelo projeto apiteste.

Como uma API Web funciona na prática

O fluxo básico de uma requisição é sempre o mesmo:

  1. O cliente (browser, app, script) monta uma requisição com método, URL e dados opcionais
  2. A requisição viaja pela internet usando o protocolo HTTP/HTTPS
  3. O servidor recebe, valida, processa e consulta o banco se necessário
  4. O servidor devolve uma resposta com status code e corpo (geralmente JSON)
  5. O cliente lê o status, processa os dados e age de acordo

Documentação oficial

Toda API bem construída possui documentação. Para a API deste projeto (FastAPI), a documentação interativa é gerada automaticamente e fica acessível em:

  • http://apiteste.codermaster.com.br/api/docs — Swagger UI (interface interativa)
  • http://apiteste.codermaster.com.br/api/redoc — ReDoc (leitura mais limpa)

Referências externas essenciais:

HTTP e Métodos

HTTP (HyperText Transfer Protocol) é o protocolo de comunicação que as APIs REST utilizam. Toda requisição tem três partes: método, URL e opcionalmente um corpo.

Anatomia de uma requisição HTTP

POST /api/v1/students HTTP/1.1
Host: apiteste.codermaster.com.br
Content-Type: application/json
Authorization: Bearer eyJhbGci...

{
  "name": "Maria Souza",
  "email": "maria@email.com"
}

E a resposta correspondente:

HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": 42,
  "name": "Maria Souza",
  "email": "maria@email.com",
  "is_active": true
}

Métodos HTTP

MétodoAçãoExemploCorpo?
GETBuscar dadosGET /studentsNão
POSTCriar recursoPOST /studentsSim
PUTSubstituir recursoPUT /students/5Sim
PATCHAtualizar parcialmentePATCH /students/5Sim
DELETERemover recursoDELETE /students/5Não
💡
PUT vs PATCH

PUT substitui o objeto inteiro — se você omitir um campo, ele vai a nulo. PATCH atualiza só os campos enviados, mantendo o resto. Use PATCH para edições parciais.

Headers importantes

HeaderPropósitoExemplo
Content-TypeFormato do corpo enviadoapplication/json
AcceptFormato esperado na respostaapplication/json
AuthorizationToken de autenticaçãoBearer <token>
X-API-KeyChave de API simplesabc123xyz

Query Parameters vs Path Parameters vs Body

Existem três formas de passar dados em uma requisição:

Path Parameter — parte da URL

# O :id é parte do caminho — identifica um recurso específico
GET /api/v1/students/42

Query Parameter — após o ?

# Filtros, paginação, buscas
GET /api/v1/students?search=maria&is_active=true&limit=20

Request Body — corpo JSON

# Dados para criar ou atualizar
POST /api/v1/students
Content-Type: application/json

{
  "name": "Maria Souza",
  "email": "maria@email.com",
  "cpf": "123.456.789-00"
}

REST e JSON

REST (Representational State Transfer) é um estilo arquitetural — um conjunto de convenções para projetar APIs HTTP. Uma API que segue REST é chamada de RESTful.

Princípios REST

  • Stateless — cada requisição é independente; o servidor não guarda estado do cliente entre chamadas
  • Recursos identificados por URLs — /students, /students/5, /students/5/enrollments
  • Verbos HTTP para ações — GET busca, POST cria, PUT/PATCH atualiza, DELETE remove
  • Representações — o mesmo recurso pode ser retornado em JSON, XML, etc.
  • Interface uniforme — convenções consistentes em toda a API

Convenções de URL

PadrãoAção
GET /studentsLista todos os alunos
POST /studentsCria um novo aluno
GET /students/{id}Retorna um aluno específico
PUT /students/{id}Substitui os dados do aluno
PATCH /students/{id}Atualiza parcialmente
DELETE /students/{id}Remove o aluno
GET /students/{id}/enrollmentsMatrículas do aluno (sub-recurso)

JSON — o formato de dados

JSON (JavaScript Object Notation) é o formato padrão para troca de dados em APIs REST. É simples, legível e suportado por todas as linguagens.

Tipos de dados em JSON

{
  "string": "texto aqui",
  "number": 42,
  "float": 9.5,
  "boolean": true,
  "null": null,
  "array": [1, 2, 3],
  "object": { "nested": "value" }
}

Exemplo real — resposta da API do projeto

// GET /api/v1/students/1
{
  "id": 1,
  "name": "João Silva",
  "email": "joao@email.com",
  "cpf": "111.222.333-44",
  "registration": "2025001",
  "birth_date": "2000-03-15",
  "phone": "(16) 99999-0001",
  "is_active": true,
  "created_at": "2025-02-10T14:30:00"
}

Lista de recursos

// GET /api/v1/students
[
  { "id": 1, "name": "João Silva", "is_active": true },
  { "id": 2, "name": "Maria Souza", "is_active": true }
]
⚠️
Atenção ao Content-Type

Sempre envie o header Content-Type: application/json ao mandar um body. Sem ele, o servidor pode rejeitar ou não interpretar corretamente os dados enviados.

Versionamento de API

Boas APIs versiona seus endpoints para que mudanças não quebrem clientes existentes. O projeto usa prefixo na URL:

# v1 — versão atual
GET /api/v1/students

# v2 — nova versão com breaking changes (hipotético)
GET /api/v2/students

Capítulo 4
Status Codes HTTP

Cada resposta HTTP tem um status code — um número de 3 dígitos que indica o resultado da requisição. Saber ler esses códigos é fundamental para depurar problemas.

Grupos de status

FaixaSignificado
2xxSucesso — a requisição foi aceita e processada
3xxRedirecionamento — o recurso foi movido
4xxErro do cliente — problema na requisição
5xxErro do servidor — problema interno

Códigos mais comuns

CódigoNomeQuando ocorre
200OKGET/PUT/PATCH/DELETE bem-sucedido
201CreatedPOST criou um novo recurso
204No ContentSucesso sem corpo (comum em DELETE)
400Bad RequestDados inválidos ou campos faltando
401UnauthorizedSem autenticação ou token inválido
403ForbiddenAutenticado mas sem permissão
404Not FoundRecurso não existe
409ConflictDado duplicado (ex: email já cadastrado)
422Unprocessable EntityValidação falhou (FastAPI usa muito)
500Internal Server ErrorBug no servidor

Exemplo de erro 422 do FastAPI

Quando você envia dados inválidos, o FastAPI devolve exatamente quais campos falharam:

// POST /api/v1/students — com campo "email" faltando
// Resposta: HTTP 422
{
  "detail": [
    {
      "loc": ["body", "email"],
      "msg": "field required",
      "type": "missing"
    }
  ]
}

Exemplo de erro 404

// GET /api/v1/students/9999 — id inexistente
// Resposta: HTTP 404
{
  "detail": "Student not found"
}
💡
Regra prática

Se o código começa com 4, o problema está no que você enviou (URL errada, dados faltando, sem autenticação). Se começa com 5, o problema é no servidor e você precisa verificar os logs do backend.

Como o projeto usa os status codes

OperaçãoStatus esperado
GET /students (lista)200 OK
GET /students/{id} (existe)200 OK
GET /students/{id} (não existe)404 Not Found
POST /students (dados válidos)201 Created
POST /students (email duplicado)400 Bad Request
PUT /students/{id}200 OK
DELETE /students/{id}200 OK (retorna objeto desativado)

Postman — Introdução e Interface

Postman é a ferramenta mais popular para testar, documentar e explorar APIs. Com ele você monta requisições HTTP visualmente, sem escrever código, e vê as respostas formatadas.

🔗
Download e documentação oficial

Download: https://www.postman.com/downloads/
Docs: https://learning.postman.com/docs/getting-started/overview/

Instalação

  1. Acesse postman.com/downloads e baixe para o seu sistema
  2. Instale normalmente (Windows: .exe, Mac: .dmg, Linux: .tar.gz ou snap)
  3. Crie uma conta gratuita ou clique em "Skip and go to the app"
  4. A interface principal abre — você já pode fazer requisições

Partes principais da interface

ÁreaO que faz
Sidebar esquerdaColeções, ambientes, histórico de requisições
Barra de URLMétodo HTTP + URL da requisição
Abas de configuraçãoParams, Headers, Body, Auth, Pre-req, Tests
Painel de respostaBody, Headers, status code, tempo de resposta
EnvironmentsVariáveis por ambiente (local, produção)

Primeira requisição

  1. Clique em "New" → "HTTP Request"
  2. Selecione o método GET no dropdown
  3. Digite a URL: http://apiteste.codermaster.com.br/api/v1/students
  4. Clique em Send
  5. Veja a resposta JSON no painel inferior

Aba Params

Para adicionar query parameters sem digitá-los na URL manualmente, use a aba Params. Cada linha vira ?chave=valor na URL automaticamente.

KeyValueResultado na URL
searchmaria?search=maria
is_activetrue?search=maria&is_active=true

Aba Headers

Headers são metadados da requisição. Os mais comuns que você vai adicionar:

KeyValue
Content-Typeapplication/json
AuthorizationBearer seu-token-aqui
Acceptapplication/json
💡
Postman já define Content-Type automaticamente

Quando você usa a aba Body com a opção "raw → JSON", o Postman já define o header Content-Type: application/json automaticamente. Você não precisa adicionar manualmente.

Capítulo 6
Postman — Fazendo Requisições

Neste capítulo vamos cobrir como executar cada tipo de requisição HTTP no Postman: GET, POST, PUT, PATCH e DELETE.

GET — Buscando dados

Não tem corpo. Só URL e parâmetros opcionais.

  1. Método: GET
  2. URL: http://apiteste.codermaster.com.br/api/v1/students
  3. Aba Params → adicione filtros se desejar (ex: is_active = true)
  4. Send → resposta 200 com lista JSON

POST — Criando um recurso

Requer um corpo JSON com os dados do novo recurso.

  1. Método: POST
  2. URL: http://apiteste.codermaster.com.br/api/v1/students
  3. Aba Body → selecione raw → dropdown JSON
  4. Cole o JSON no editor: 
    {
  "name": "Ana Lima",
  "email": "ana@email.com",
  "cpf": "999.888.777-66",
  "birth_date": "2001-05-20",
  "phone": "(16) 98888-0002"
}
  5. Send → resposta 201 com o objeto criado (incluindo o id gerado)

PUT — Substituição completa

  1. Método: PUT
  2. URL: http://apiteste.codermaster.com.br/api/v1/students/1
  3. Body → raw → JSON com todos os campos do aluno
  4. Send → resposta 200 com objeto atualizado

PATCH — Atualização parcial

Envie apenas os campos que deseja alterar:

// PATCH /api/v1/students/1
{
  "phone": "(16) 97777-9999"
}

DELETE — Removendo

  1. Método: DELETE
  2. URL: http://apiteste.codermaster.com.br/api/v1/students/1
  3. Sem body necessário
  4. Send → resposta 200 (aluno desativado no projeto)

Lendo a resposta

O painel de resposta tem várias abas úteis:

AbaO que mostra
BodyO JSON retornado — use "Pretty" para formatar
HeadersHeaders da resposta (Content-Type, etc.)
StatusCódigo HTTP (200, 201, 404...)
TimeTempo da requisição em ms
SizeTamanho da resposta
⚠️
Erro de conexão vs erro HTTP

Se o Postman mostrar "Could not get any response", o servidor não está rodando ou a URL está errada. Se mostrar um código 4xx ou 5xx, o servidor respondeu com erro — leia o body para entender o motivo.

Histórico de requisições

O Postman salva automaticamente todas as requisições em History (sidebar esquerda). Você pode clicar em qualquer requisição anterior para reabrí-la. Para salvar permanentemente, adicione a uma coleção clicando em Save.

Capítulo 7
Postman — Coleções e Ambientes

Coleções e Ambientes são os dois recursos mais importantes do Postman para organizar o trabalho com APIs de verdade.

Coleções

Uma coleção é uma pasta que agrupa requisições relacionadas. É o equivalente a um projeto — você organiza todas as chamadas de uma API em uma coleção e compartilha com o time.

Criando uma coleção

  1. Sidebar → clique em Collections
  2. Clique no + ou "New Collection"
  3. Nomeie: ex. apiteste — Escola
  4. Clique em Create

Adicionando requisições à coleção

  1. Monte uma requisição normalmente
  2. Clique em Save (Ctrl+S)
  3. Escolha a coleção destino e dê um nome à requisição
  4. Ela aparece na sidebar organizada

Pastas dentro de coleções

Crie subpastas por domínio: Students, Teachers, Subjects, etc. Fica muito mais fácil navegar em projetos grandes.

Ambientes (Environments)

Ambientes guardam variáveis que mudam conforme o contexto. O exemplo clássico: a URL base muda entre local e produção.

Criando um ambiente

  1. Clique no ícone de olho (canto superior direito) → Add
  2. Nomeie: Local
  3. Adicione variáveis:
    VariableInitial Value
    base_urlhttp://apiteste.codermaster.com.br/api/v1
    token(deixe vazio por enquanto)
  4. Clique em Save
  5. Selecione o ambiente no dropdown no topo direito

Usando variáveis nas requisições

Use a sintaxe {{nome_da_variavel}} em qualquer campo:

GET {{base_url}}/students
GET {{base_url}}/students/{{student_id}}

Quando você tiver ambientes Local e Producao, basta trocar o ambiente no dropdown e todas as URLs se atualizam automaticamente.

Scripts de teste (Tests)

A aba Tests permite escrever scripts JavaScript que rodam após cada resposta:

// Verifica status 200
pm.test("Status é 200", () => {
  pm.response.to.have.status(200);
});

// Verifica que retornou um array
pm.test("Retornou lista", () => {
  const body = pm.response.json();
  pm.expect(body).to.be.an("array");
});

// Salva o id do primeiro aluno como variável
const first = pm.response.json()[0];
pm.environment.set("student_id", first.id);
💡
Collection Runner

Com o Collection Runner você executa toda a coleção em sequência automaticamente — ótimo para testar todos os endpoints de uma vez. Acesse via botão "Run" na coleção.

Importar e Exportar coleções

Coleções podem ser exportadas como arquivo JSON e compartilhadas com o time. Para importar: Import → arrastar o arquivo .json. O Postman também suporta importar diretamente de uma URL de spec OpenAPI/Swagger — o FastAPI gera essa spec em /openapi.json.

Python Requests — Fundamentos

A biblioteca requests é o jeito mais simples de fazer chamadas HTTP em Python. Com ela você pode automatizar testes, integrar sistemas e consumir APIs sem abrir o Postman.

📦
Instalação

pip install requests
Documentação oficial: https://requests.readthedocs.io

GET — Buscando dados

import requests

BASE = "http://apiteste.codermaster.com.br/api/v1"

# Lista todos os alunos
response = requests.get(f"{BASE}/students")

print(response.status_code)  # 200
print(response.json())       # lista Python

# Com filtro por query param
response = requests.get(
    f"{BASE}/students",
    params={"is_active": True, "search": "maria"}
)

# Buscar por id
response = requests.get(f"{BASE}/students/1")
aluno = response.json()
print(aluno["name"])

POST — Criando um recurso

novo_aluno = {
    "name": "Beatriz Costa",
    "email": "beatriz@email.com",
    "cpf": "555.444.333-22",
    "birth_date": "2002-07-10",
}

response = requests.post(
    f"{BASE}/students",
    json=novo_aluno  # define body + Content-Type automaticamente
)

if response.status_code == 201:
    criado = response.json()
    print(f"Criado com id {criado['id']}")
else:
    print(f"Erro {response.status_code}: {response.text}")

PUT e PATCH — Atualizando

# PUT — substitui tudo
response = requests.put(
    f"{BASE}/students/1",
    json={
        "name": "João Silva Atualizado",
        "email": "joao.novo@email.com",
        "cpf": "111.222.333-44",
        "birth_date": "2000-03-15",
    }
)

# PATCH — atualiza só o que você manda
response = requests.patch(
    f"{BASE}/students/1",
    json={"phone": "(16) 96666-7777"}
)

DELETE — Removendo

response = requests.delete(f"{BASE}/students/1")
print(response.status_code)  # 200

Tratamento de erros

response = requests.get(f"{BASE}/students/9999")

# Forma simples — lança exceção se 4xx ou 5xx
response.raise_for_status()  # HTTPError se erro

# Forma manual
if response.status_code == 404:
    print("Aluno não encontrado")
elif response.status_code == 422:
    erros = response.json()["detail"]
    for e in erros:
        print(f"Campo {e['loc']}: {e['msg']}")

Enviando headers

headers = {
    "Authorization": "Bearer meu-token",
    "Accept": "application/json"
}

response = requests.get(
    f"{BASE}/students",
    headers=headers
)

Capítulo 10
Python Requests — Uso Avançado

Neste capítulo avançamos para padrões profissionais: sessões, timeouts, tratamento robusto de erros e um script completo integrando com o projeto apiteste.

Session — reutilizando conexões

O objeto Session mantém headers e cookies entre requisições e reutiliza conexões TCP — muito mais eficiente para múltiplas chamadas.

session = requests.Session()
session.base_url = "http://apiteste.codermaster.com.br/api/v1"

# Headers que todas as requisições vão usar
session.headers.update({
    "Content-Type": "application/json",
    "Accept": "application/json",
})

# Agora toda requisição da session carrega esses headers
r = session.get("/students")  # (relativo — combine com base_url)

Timeout

Sempre defina timeout para não travar o script esperando uma resposta que nunca vem:

import requests
from requests.exceptions import Timeout, ConnectionError

try:
    r = requests.get(
        "http://apiteste.codermaster.com.br/api/v1/students",
        timeout=(3, 10)  # (connect_timeout, read_timeout)
    )
except Timeout:
    print("A requisição demorou demais")
except ConnectionError:
    print("Servidor não está disponível")

Script completo — fluxo de matrícula

Este script cria todo o fluxo do projeto: período → professor → disciplina → aluno → matrícula → nota.

import requests

BASE = "http://apiteste.codermaster.com.br/api/v1"
s = requests.Session()
s.headers["Content-Type"] = "application/json"

def post(path, data):
    r = s.post(f"{BASE}{path}", json=data)
    r.raise_for_status()
    return r.json()

# 1. Período
periodo = post("/periods", {
    "name": "2025/1", "year": 2025, "semester": 1,
    "start_date": "2025-02-01", "end_date": "2025-06-30",
    "is_active": True
})

# 2. Professor
professor = post("/teachers", {
    "name": "Prof. Ana", "email": "ana@escola.com",
    "hire_date": "2020-01-10"
})

# 3. Disciplina
disciplina = post("/subjects", {
    "name": "Física", "code": "FIS101",
    "workload_hours": 60, "teacher_id": professor["id"]

from requests.exceptions import Timeout, ConnectionError as ConnErr

def api_get(path, params=None):
    try:
        r = s.get(f"{BASE}{path}", params=params, timeout=(5, 15))
        if r.status_code == 200:
            return r.json()
        elif r.status_code == 404:
            print(f"Não encontrado: {path}")
        elif r.status_code >= 500:
            print(f"Erro no servidor: {r.status_code}")
        return None
    except Timeout:
        print("Timeout — servidor demorou demais")
    except ConnErr:
        print("Sem conexão com o servidor")

    media_geral = sum(medias) / len(medias)
    print(f"{disc:20s} | média {media_geral:.1f} | {aprovados}/{len(rows)} aprovados")
